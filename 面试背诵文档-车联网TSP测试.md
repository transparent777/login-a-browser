## 二、核心概念速记表（面试必问）

### 2.1 核心术语

| 术语 | 全称 / 含义 | 一句话解释 |
|------|------------|-----------|
| **TSP** | Telematics Service Platform | 车联网服务平台，整个云端的核心 |
| **TBOX** | Telematics Box | 装在车上的智能终端，负责采集车辆数据 + 4G 通信 |
| **CAN 总线** | Controller Area Network | 车内部件之间的通信总线，TBOX 从这里读数据 |
| **ECU** | Electronic Control Unit | 电子控制单元，车上的各个控制模块 |
| **OTA** | Over-The-Air | 远程升级，通过无线方式给车辆升级固件/软件 |
| **MQTT** | Message Queuing Telemetry Transport | 轻量级物联网通信协议，TBOX 和平台之间的主要通信方式 |
| **VIN** | Vehicle Identification Number | 车辆识别码，17 位，每辆车唯一 |
| **ICCID** | Integrated Circuit Card ID | SIM 卡唯一标识，TBOX 里的 SIM 卡编号 |
| **GNSS** | Global Navigation Satellite System | 全球导航卫星系统，GPS/北斗都属于 GNSS |

### 2.2 国标和行业标准

| 标准 | 全称 / 用途 | 关键点 |
|------|------------|--------|
| **GB/T 32960** | 电动汽车远程服务与管理系统技术规范 | 新能源车必须遵守，数据项包括：整车数据、驱动电机数据、燃料电池数据、发动机数据、车辆位置数据、极值数据、报警数据 |
| **JT/T 808** | 道路运输车辆卫星定位系统终端通讯协议 | 商用车（货车、客车）用的通信协议 |
| **JT/T 1078** | 道路运输车辆卫星定位系统视频通讯协议 | 视频监控相关 |
| **GB 17691** | 重型柴油车污染物排放标准 | 国六排放，涉及排放数据远程监控 |

### 2.3 GB/T 32960 上报流程（面试高频）

```
车辆启动/充电 → TBOX 采集数据 → 每 30 秒打包上报一次实时数据
    → TSP 平台接收后存储 + 转发到国家平台
    → 国家平台返回接收确认
```

**三要素**：
1. **登录认证** — TBOX 使用 VIN 作为唯一标识向平台注册
2. **实时上报** — 行驶中每 30 秒上报一次，停驶后可以降低频率
3. **补发机制** — 通信中断恢复后，TBOX 需要补发离线期间的存储数据

## 五、技术基础 — 必须能说上来

### 5.1 HTTP 协议（接口测试基础）

| 面试常见问 | 回答要点 |
|-----------|----------|
| HTTP 和 HTTPS 区别 | HTTPS = HTTP + SSL/TLS 加密，端口 443 vs 80 |
| GET 和 POST 区别 | GET 参数在 URL 中，有长度限制；POST 参数在 Body 中，适合提交大量数据 |
| 常见状态码 | 200 成功、201 创建成功、400 参数错误、401 未认证、403 无权限、404 不存在、500 服务器错误、502 网关错误 |
| RESTful API | 用 GET/POST/PUT/DELETE 操作资源（增删改查） |
| Token 鉴权 | 登录后获取 Token，后续请求在 Header 中带 `Authorization: Bearer xxx` |

### 5.2 MQTT 协议（TBOX 通信核心 — 必问！）

```
MQTT 是发布/订阅模式的轻量级物联网协议，核心优势：低带宽、低功耗、支持不稳定网络

┌─────────────────────────────────────────┐
│              MQTT Broker                 │
│        (mosquitto / EMQX)               │
│                                          │
│   Topic: /vehicle/{VIN}/data            │
│   Topic: /vehicle/{VIN}/control         │
│   Topic: /vehicle/{VIN}/alert           │
│   Topic: /vehicle/{VIN}/ota             │
└──────┬──────────────┬───────────────────┘
       │ Subscribe    │ Subscribe
       ▼              ▼
  ┌─────────┐   ┌──────────┐
  │  TBOX   │   │TSP 平台  │
  │Publish  │   │Publish   │
  │Subscribe│   │Subscribe  │
  └─────────┘   └──────────┘
```

| 必背术语 | 解释 |
|---------|------|
| **Topic（主题）** | 消息的分类标签，如 `/vehicle/VIN123/data` |
| **QoS（服务质量）** | QoS 0：最多发一次（可能丢失）；QoS 1：至少发一次（可能重复）；QoS 2：恰好发一次（最安全但最慢） |
| **Retain（保留消息）** | Broker 会保存最后一条保留消息，新订阅者立即收到 |
| **Will Message（遗嘱消息）** | 客户端异常断开时 Broker 自动发布的预设消息 |
| **Keep Alive** | 心跳间隔，TBOX 一般设为 60-120 秒 |

### 5.3 TCP 协议

| 要点 | 说明 |
|------|------|
| 三次握手 | SYN → SYN-ACK → ACK，建立连接 |
| 四次挥手 | FIN → ACK → FIN → ACK，断开连接 |
| TCP vs UDP | TCP 可靠、有序、有连接；UDP 快、无连接、可能丢包 |
| 为什么车联网用 TCP | 车辆数据要求可靠传输，不能丢 |

## 九、常见面试追问 + 参考回答

### Q1：你了解 MQTT 吗？它和 HTTP 有什么区别？
> **答**：MQTT 是发布/订阅模式的物联网协议，特点是轻量级、低带宽、支持心跳保活，非常适合车载终端这种网络不稳定的场景。和 HTTP 相比，HTTP 是请求/响应模式，每次都得客户端主动拉；而 MQTT 支持服务端主动推送，TSP 平台可以随时给 TBOX 下发指令。车联网里一般用 MQTT 做 TBOX 和平台之间的长连接通信，用 HTTP 做 APP 和后端的接口调用。

### Q2：如果远程控制指令下发后没反应，你怎么排查？
> **答**：我会按照链路逐层排查：① 先确认车辆在线状态，TBOX 是否断连；② 看平台日志，确认指令是否成功推送到 MQTT Topic；③ 用 MQTT 客户端工具订阅对应 Topic，确认消息是否到达 Broker；④ 看 TBOX 日志（如果可获取），确认是否收到指令、是否执行；⑤ 检查指令超时策略是否生效，是否有重试记录；⑥ 抓包分析 MQTT 报文，确认通信链路正常。

### Q3：测试环境和生产环境有什么区别需要注意？
> **答**：测试环境可能只有几台测试车，网络条件好，但生产环境有几十万台车并发上报，网络条件复杂（隧道、高速、地下室）。所以我测试时会额外关注：① 弱网测试（模拟丢包、高延迟）；② 并发压力测试（模拟大量 TBOX 同时上报）；③ 数据量大时的查询性能（如轨迹回放加载几千个点）。

### Q4：你了解 GB/T 32960 吗？
> **答**：了解。这是国家强制标准，要求新能源车必须把运行数据上报给国家监管平台。核心数据包括整车数据（车速、里程、档位）、驱动电机数据、电池数据（SOC、温度、电压）、车辆位置等。采集频率通常是行驶时 30 秒一次。测试时重点关注：数据项是否完整、时间戳是否正确、位置漂移是否合理、上报频率是否符合要求。

### Q5：说一个你遇到过的测试难点？
> **答**：（结合你做项目的经历说）我在做 TSP 模拟测试项目时，遇到过 MQTT 消息乱序接收的问题。我用 Python 脚本模拟 TBOX 发了 100 条顺序数据，但平台接收时顺序打乱了。排查发现是 QoS 设置了 0，后来改为 QoS 1 并给每条消息增加了序列号字段，在数据库端按序列号重新排序后再处理。这让我理解了在物联网场景下消息可靠性很重要。
